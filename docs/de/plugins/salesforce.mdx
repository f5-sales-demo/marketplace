Das **salesforce**-Plugin integriert die Salesforce CLI mit Claude Code,
bietet Org-Authentifizierung und einen universellen CLI-Agenten, der
beliebige `sf`-Befehle einschließlich SOQL-Abfragen ausführen kann. Es unterstützt
Headless-Container-Umgebungen mit JWT-, Access-Token- und SFDX-URL-
Authentifizierungsabläufen. Der nachstehende Verwendungsleitfaden zeigt, wie Sie
natürliche Sprache nutzen können, um Konten, Opportunities, Cases und Contacts
über den CLI-Agenten abzufragen.

<Badge text="v1.0.0" variant="note" /> <Badge text="Development" variant="tip" />

## Installation

```
/plugin install salesforce@f5-sales-demo-marketplace
```

## Voraussetzungen

- **Salesforce CLI** (`@salesforce/cli`)
- **Salesforce-Org** mit API-Zugriff
- Ein Benutzerkonto mit Berechtigungen zum Abfragen von Objekten (Cases, Opportunities, Accounts, Contacts)

Installieren Sie die Salesforce CLI auf Ihrer Arbeitsstation:

```bash
brew install sf
```

Überprüfen Sie die Installation:

```bash
sf --version
```

Die Ausgabe sollte in etwa so aussehen: `@salesforce/cli/2.x.x`.

## Authentifizierungseinrichtung

### Schritt 1: Ihre Salesforce-Domain finden

Schauen Sie in Ihrem Browser auf Ihre Salesforce-URL. Wenn Sie Salesforce unter
`https://acme.lightning.force.com` aufrufen, ist Ihre Login-Domain
`https://acme.my.salesforce.com`.

<Aside type="tip">
  Wenn Ihr Unternehmen SSO verwendet, müssen Sie Ihre benutzerdefinierte Domain mit
  `--instance-url` angeben, damit die Anmeldung über Ihren SSO-Anbieter
  und nicht über die generische Salesforce-Anmeldeseite umgeleitet wird.
</Aside>

### Schritt 2: Authentifizierung über den Browser

```bash
sf org login web --alias my-org --set-default --instance-url https://YOUR-DOMAIN.my.salesforce.com
```

Ersetzen Sie `YOUR-DOMAIN` durch Ihre Unternehmens-Domain aus Schritt 1. Ihr Browser
öffnet sich — schließen Sie den SSO- oder Anmeldevorgang ab und autorisieren Sie die App. Sie sollten
Folgendes sehen:

```
Successfully authorized your-email@company.com with org ID 00DXXXXXXXXXXXXXXX
```

### Schritt 3: Verbindung überprüfen

```bash
sf org display --target-org my-org
```

Bestätigen Sie, dass die Ausgabe `Connected Status: Connected` mit Ihrem Benutzernamen
und der Instanz-URL anzeigt.

### Schritt 4: Portable Authentifizierung für Container (optional)

Wenn Sie sich in einer Headless-Umgebung (CI/CD, Container,
Remote-Sitzung) authentifizieren müssen, exportieren Sie die SFDX-Auth-URL von Ihrer Arbeitsstation:

```bash
sf org display --verbose --target-org my-org
```

Kopieren Sie den Wert der **Sfdx Auth Url** (beginnt mit `force://`). Führen Sie im
Container Folgendes aus:

```bash
export SFDX_AUTH_URL="force://PlatformCLI::YOUR_AUTH_TOKEN@your-domain.my.salesforce.com"
echo "$SFDX_AUTH_URL" | sf org login sfdx-url --sfdx-url-stdin=- --alias=my-org --set-default
```

<Aside type="caution">
  Die SFDX-Auth-URL ist ein Anmeldedatum — behandeln Sie sie wie ein Passwort. Speichern Sie
  sie nicht in Dateien und fügen Sie sie nicht in Chats ein. Verwenden Sie ausschließlich
  Umgebungsvariablen und stdin-Pipes.
</Aside>

### Referenz der Authentifizierungsmethoden

| Methode | Geeignet für | Erfordert |
|--------|----------|----------|
| Web-Login | Arbeitsstationen mit Browser | Browser + SSO |
| SFDX-URL | Container, CI/CD, portable Authentifizierung | Auth-URL aus einer authentifizierten Sitzung |
| JWT Bearer | Automatisierte Pipelines | Connected App + privater Schlüssel + Consumer Key |
| Access Token | Authentifizierung über Umgebungsvariable | `SF_ACCESS_TOKEN` + `SF_ORG_INSTANCE_URL` |

**Hinweis:** Der Geräteablauf (`sf org login device`) ist seit August
2025 gesperrt und wird nicht unterstützt.

### Umgebungsvariablen

| Variable | Zweck |
|----------|---------|
| `SF_ACCESS_TOKEN` | Bearer-Token für Access-Token-Authentifizierung |
| `SFDX_AUTH_URL` | Force-Auth-URL für SFDX-URL-Authentifizierung |
| `SF_ORG_INSTANCE_URL` | Org-Instanz-URL |
| `SF_JWT_KEY_FILE` | Pfad zum privaten JWT-Schlüssel |
| `SF_CLIENT_ID` | Consumer Key der Connected App |
| `SF_USERNAME` | Salesforce-Benutzername für JWT |

## Befehle

### /salesforce:sf-status

Überprüfen Sie den Verbindungsstatus Ihrer Salesforce-Org, die CLI-Version und
authentifizierte Orgs.

```
/salesforce:sf-status
```

**Was zu erwarten ist:** Eine Tabelle mit Ihrer sf CLI-Version, dem authentifizierten
Org-Alias, Benutzernamen, Instanz-URL, Verbindungsstatus und API-Version.

### /salesforce:sf-login

Authentifizieren Sie sich bei einer Salesforce-Org. Optional können Sie einen Alias angeben.

```
/salesforce:sf-login
/salesforce:sf-login my-org
```

**Was zu erwarten ist:** Wenn eine Org mit diesem Alias bereits authentifiziert ist,
werden die Verbindungsdetails angezeigt. Andernfalls werden verfügbare
Anmeldedaten (Umgebungsvariablen) geprüft und die Authentifizierung mit der ersten
vollständig erfüllten Methode durchgeführt.

## Skills

### salesforce-index

Übergeordneter Intent-Router. Wird automatisch aktiviert, wenn Sie
Salesforce, Org-Verwaltung oder Datenabfragen in natürlicher Sprache erwähnen.

| Sie sagen | Weiterleitung an |
|---------|-----------|
| „bei salesforce anmelden", „org authentifizieren" | `salesforce-auth`-Skill |
| „org-status prüfen", „orgs auflisten" | `cli-operator`-Agent |
| „zeig mir opportunities", „case nachschlagen" | `cli-operator`-Agent (SOQL-Abfrage) |
| „Apex-Klasse generieren", „Flow erstellen" | afv-library-Entwicklungs-Skills |

### salesforce-auth

Container-angepasster Authentifizierungs-Skill. Unterstützt vier Authentifizierungsmethoden und
erkennt automatisch, welche Anmeldedaten verfügbar sind. Priorisierung:
Access-Token, dann JWT, dann SFDX-URL, dann Web-Login.

## Agenten

### cli-operator

Autonomer Salesforce CLI-Agent, der `sf`-Befehle mit Sicherheitsschranken ausführt.
Alle Salesforce-Operationen werden an diesen Agenten delegiert, um den
Hauptsitzungskontext schlank zu halten.

**Sicherheitsregeln:**

- Standardmäßig nur lesend — Schreiboperationen erfordern explizite Genehmigung
- Keine Bereitstellung ohne Vorschau und Bestätigungsschritt
- Gibt niemals Access-Token, Auth-URLs oder Anmeldedaten in der Ausgabe aus
- Bereinigt alle vom Benutzer bereitgestellten Werte (lehnt Shell-Metazeichen ab)
- Verwendet `--json`-Ausgabe für strukturierte, parsierbare Ergebnisse

**Antwortformat:**

```
## Result: [SUCCESS | FAILURE | PARTIAL]
### Command Executed
### Output Summary
### Issues
```

## Verwendungsleitfaden: Kontoverwaltung

<Aside type="note">
  Die nachstehenden Eingabeaufforderungen sind keine fest codierten Funktionen des Plugins. Sie funktionieren,
  weil der cli-operator-Agent beliebige `sf` CLI-Befehle ausführen kann, einschließlich
  beliebiger SOQL-Abfragen. Die Ergebnisse hängen vom Datenmodell Ihrer Org und
  Ihren Benutzerberechtigungen ab. Die hier aufgeführten Eingabeaufforderungen wurden getestet und verifiziert,
  um mit Standard-Salesforce-Objekten zu funktionieren (Account, Contact, Case,
  Opportunity, AccountTeamMember, OpportunityTeamMember).
</Aside>

Diese Eingabeaufforderungen funktionieren mit jeder Salesforce-Org. Ersetzen Sie die Platzhalterwerte
durch Ihre eigenen Informationen. Jede Eingabeaufforderung liefert Ergebnisse, die spezifisch
für Ihre Rolle, Konten und Ihr Vertriebsgebiet sind.

<Aside type="tip">
  Für beste Ergebnisse geben Sie Ihre E-Mail-Adresse in Eingabeaufforderungen an, bei denen
  Ihr Benutzerdatensatz identifiziert werden muss. Salesforce-Orgs können mehrere Benutzer
  mit ähnlichen Namen haben.
</Aside>

### Erste Schritte

Überprüfen Sie nach der Authentifizierung, ob alles funktioniert:

```
/salesforce:sf-status
```

Versuchen Sie dann eine einfache natürlichsprachliche Abfrage:

```
list my authenticated salesforce orgs
```

**Was zu erwarten ist:** Eine Tabelle mit allen verbundenen Orgs mit Alias,
Benutzernamen, Instanz-URL und Verbindungsstatus.

### Ihre Konten finden

Ermitteln Sie, welchen Konten Sie zugewiesen sind:

```
what salesforce accounts am I on the account team for? My email is your-email@company.com
```

**Was zu erwarten ist:** Eine Liste von Konten, gruppiert nach Kontoinhaber, mit Ihrer
Team-Mitgliederrolle für jedes Konto. Wenn die meisten Rollen als leer angezeigt werden, ist das
eine häufige Datenhygienelücke — die Kontoteam-Mitgliedschaften sind vorhanden, aber die Rollen
wurden nicht befüllt.

### Abdeckung mit einem Kollegen abgleichen

Vergleichen Sie Ihre Kontoabdeckung mit einem Teammitglied, um Lücken zu finden:

```
In salesforce, find all accounts where Colleague Name is on the account team. Then for each of those accounts, check if your-email@company.com is also on the account team. Show me two lists: accounts where we are BOTH tagged, and accounts where only my colleague is tagged but I am missing.
```

**Was zu erwarten ist:** Eine Übersichtstabelle mit Überschneidungsanzahl und Lückenanzahl,
gefolgt von zwei Listen. Wenn Sie kürzlich die Rolle gewechselt haben, finden Sie möglicherweise
keine Überschneidungen — dies zeigt, welche Konten Sie hinzugefügt werden müssen.

<Aside type="note">
  Sie können auch nach einem Vorgänger suchen, indem Sie Folgendes hinzufügen: „Prüfen Sie auch, ob
  Vorheriger Personenname auf einem dieser Konten erscheint." Dies hilft,
  Übergabelücken zu identifizieren, wenn jemand in einem Vertriebsgebiet ersetzt wird.
</Aside>

### Pipeline-Überblick für das Vertriebsgebiet

Sehen Sie alle offenen Opportunities für Ihre Konten:

```
show me all open salesforce opportunities on Colleague Name's account team accounts, sorted by close date, include the stage, amount, and probability
```

**Was zu erwarten ist:** Eine Zusammenfassung mit der Gesamtanzahl der Opportunities, der ungewichteten
Pipeline und der gewichteten Pipeline. Dann eine Phasenverteilungstabelle und eine Liste der
wichtigsten Opportunities nach Betrag. Die Ausgabe kennzeichnet Datenhygieneprobleme wie
überfällige Abschlussdaten oder fehlende Beträge.

### Opportunity-Detailanalyse

Erhalten Sie ein vollständiges Bild zu einem bestimmten Deal:

```
show me a detailed view of the OPPORTUNITY NAME opportunity in salesforce - include the opportunity team members, any activities or tasks, and the account contacts
```

**Was zu erwarten ist:** Die Opportunity-Übersicht (Phase, Betrag, Abschlussdatum,
Wahrscheinlichkeit, Prognose-Kategorie, Inhaber), Opportunity-Teammitglieder mit
Rollen, verknüpfte Contacts, aktuelle Tasks und Ereignisse sowie die Feldänderungshistorie,
die zeigt, wie sich Dealvolumen und Abschlussdatum im Laufe der Zeit verändert haben.

### Support-Cases im Vertriebsgebiet

Prüfen Sie offene Support-Cases für Ihre Konten:

```
show me all open salesforce cases across Colleague Name's account team accounts, grouped by account, sorted by most recent first
```

**Was zu erwarten ist:** Eine Anzahl offener Cases mit Prioritäts- und Statusaufschlüsselung,
dann Case-Details gruppiert nach Konto. Veraltete Cases (Monate oder
Jahre alt ohne Aktivität) werden als Hygienekandidaten gekennzeichnet.

### Quartalsweise Pipeline-Prognose

Erhalten Sie eine prognosegeeignete Ansicht Ihrer Pipeline:

```
for Colleague Name's accounts, show me a quarterly pipeline summary - group the open opportunities by close date quarter with count, total amount, and weighted amount for each quarter
```

**Was zu erwarten ist:** Eine Tabelle mit einer Zeile pro Quartal, die Opportunity-Anzahl,
Gesamt-Pipeline und gewichtete Pipeline anzeigt. Enthält eine Phasenmischungsaufschlüsselung
pro Quartal und hebt die wichtigsten gewichteten Deals hervor. Überfällige
Opportunities werden separat gruppiert.

### Vollständiger Kontoüberblick

Detaillierte Analyse eines bestimmten Kunden:

```
give me a full account overview for ACCOUNT NAME in salesforce including contacts, open opportunities, and recent cases
```

**Was zu erwarten ist:** Unternehmensprofil (Branche, Umsatz, Mitarbeiterzahl,
Standort), wichtige Contacts mit Titeln und E-Mail, offene Opportunities mit
Phasen und Beträgen sowie aktuelle Support-Cases mit Status.

### Case-Nachschlagen

Suchen Sie nach einem bestimmten Support-Case anhand der Nummer:

```
look up salesforce case CASE-NUMBER and show me the case details, the customer account, and who owns it
```

**Was zu erwarten ist:** Case-Betreff, Status, Priorität, Beschreibung, das
Kundenkontoprofil und der Case-Inhaber mit Kontaktinformationen.

## Verwendungsleitfaden: Datenabfragen

Erfahrene Benutzer können spezifische SOQL-artige Abfragen in natürlicher
Sprache stellen. Das Plugin übersetzt Ihre Anfrage in das entsprechende SOQL
und führt es aus.

### Spezifische Felder

```
query salesforce for all Contacts at ACCOUNT NAME - show Name, Title, Email, Phone, and Department
```

### Datumsfilterung

```
show me all salesforce opportunities that closed won in the last 90 days on ACCOUNT NAME
```

### Aggregation

```
count all open salesforce cases grouped by priority and status across my accounts
```

### Benutzerdefinierte Objekte

```
query the CUSTOM_OBJECT__c object in salesforce for records where Status__c = 'Active'
```

<Aside type="tip">
  Wenn eine Abfrage fehlschlägt, weil ein Feld nicht existiert, wiederholt das Plugin
  die Abfrage automatisch ohne dieses Feld. Benutzerdefinierte Felder in Ihrer Org
  können andere Namen verwenden — überprüfen Sie Ihr Salesforce-Setup, wenn ein Feld
  nicht gefunden wird.
</Aside>
