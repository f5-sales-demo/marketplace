**salesforce** प्लगइन Salesforce CLI को Claude Code के साथ एकीकृत करता है,
जो org प्रमाणीकरण और एक सामान्य-उद्देश्यीय CLI एजेंट प्रदान करता है जो
SOQL क्वेरी सहित कोई भी `sf` कमांड चला सकता है। यह JWT, access-token, और SFDX URL
प्रमाणीकरण प्रवाहों के साथ हेडलेस कंटेनर वातावरणों का समर्थन करता है। नीचे दी गई
उपयोग मार्गदर्शिका दर्शाती है कि CLI एजेंट के माध्यम से प्राकृतिक भाषा का उपयोग करके
accounts, opportunities, cases, और contacts की क्वेरी कैसे करें।

<Badge text="v1.0.0" variant="note" /> <Badge text="Development" variant="tip" />

## स्थापना

```
/plugin install salesforce@f5xc-salesdemos-marketplace
```

## पूर्वापेक्षाएँ

- **Salesforce CLI** (`@salesforce/cli`)
- **Salesforce org** जिसमें API एक्सेस हो
- एक उपयोगकर्ता खाता जिसके पास ऑब्जेक्ट्स (Cases, Opportunities, Accounts, Contacts) की क्वेरी करने की अनुमतियाँ हों

अपने वर्कस्टेशन पर Salesforce CLI स्थापित करें:

```bash
brew install sf
```

स्थापना सत्यापित करें:

```bash
sf --version
```

आपको `@salesforce/cli/2.x.x` जैसा आउटपुट दिखना चाहिए।

## प्रमाणीकरण सेटअप

### चरण 1: अपना Salesforce डोमेन खोजें

ब्राउज़र में अपना Salesforce URL देखें। यदि आप Salesforce को
`https://acme.lightning.force.com` पर एक्सेस करते हैं, तो आपका लॉगिन डोमेन
`https://acme.my.salesforce.com` है।

<Aside type="tip">
  यदि आपकी कंपनी SSO का उपयोग करती है, तो आपको `--instance-url` के साथ अपना
  कस्टम डोमेन निर्दिष्ट करना होगा ताकि लॉगिन जेनेरिक Salesforce लॉगिन पेज के
  बजाय आपके SSO प्रदाता के माध्यम से रीडायरेक्ट हो।
</Aside>

### चरण 2: ब्राउज़र के माध्यम से प्रमाणीकरण करें

```bash
sf org login web --alias my-org --set-default --instance-url https://YOUR-DOMAIN.my.salesforce.com
```

`YOUR-DOMAIN` को चरण 1 से अपनी कंपनी के डोमेन से बदलें। आपका ब्राउज़र
खुलता है — SSO या लॉगिन प्रवाह पूरा करें और ऐप को अधिकृत करें। आपको
यह दिखना चाहिए:

```
Successfully authorized your-email@company.com with org ID 00DXXXXXXXXXXXXXXX
```

### चरण 3: कनेक्शन सत्यापित करें

```bash
sf org display --target-org my-org
```

पुष्टि करें कि आउटपुट आपके उपयोगकर्ता नाम और इंस्टेंस URL के साथ
`Connected Status: Connected` दिखाता है।

### चरण 4: कंटेनरों के लिए पोर्टेबल प्रमाणीकरण (वैकल्पिक)

यदि आपको हेडलेस वातावरण (CI/CD, कंटेनर, रिमोट सेशन) में प्रमाणीकरण
करने की आवश्यकता है, तो अपने वर्कस्टेशन से SFDX auth URL निर्यात करें:

```bash
sf org display --verbose --target-org my-org
```

**Sfdx Auth Url** मान कॉपी करें (जो `force://` से शुरू होता है)। कंटेनर
में, चलाएँ:

```bash
export SFDX_AUTH_URL="force://PlatformCLI::YOUR_AUTH_TOKEN@your-domain.my.salesforce.com"
echo "$SFDX_AUTH_URL" | sf org login sfdx-url --sfdx-url-stdin=- --alias=my-org --set-default
```

<Aside type="caution">
  SFDX auth URL एक क्रेडेंशियल है — इसे पासवर्ड की तरह संभालें। इसे
  फ़ाइलों में कमिट न करें या चैट में पेस्ट न करें। केवल एनवायरनमेंट
  वेरिएबल्स और stdin पाइप्स का उपयोग करें।
</Aside>

### प्रमाणीकरण विधियों का संदर्भ

| विधि | सर्वोत्तम उपयोग | आवश्यकताएँ |
|--------|----------|----------|
| Web Login | ब्राउज़र वाले वर्कस्टेशन | ब्राउज़र + SSO |
| SFDX URL | कंटेनर, CI/CD, पोर्टेबल प्रमाणीकरण | प्रमाणीकृत सेशन से Auth URL |
| JWT Bearer | स्वचालित पाइपलाइन | Connected App + प्राइवेट की + कंज्यूमर की |
| Access Token | एनवायरनमेंट वेरिएबल प्रमाणीकरण | `SF_ACCESS_TOKEN` + `SF_ORG_INSTANCE_URL` |

**नोट:** Device flow (`sf org login device`) अगस्त 2025 से अवरुद्ध है
और समर्थित नहीं है।

### एनवायरनमेंट वेरिएबल्स

| वेरिएबल | उद्देश्य |
|----------|---------|
| `SF_ACCESS_TOKEN` | access-token प्रमाणीकरण के लिए Bearer टोकन |
| `SFDX_AUTH_URL` | SFDX URL प्रमाणीकरण के लिए Force auth URL |
| `SF_ORG_INSTANCE_URL` | Org इंस्टेंस URL |
| `SF_JWT_KEY_FILE` | JWT प्राइवेट की का पथ |
| `SF_CLIENT_ID` | Connected App कंज्यूमर की |
| `SF_USERNAME` | JWT के लिए Salesforce उपयोगकर्ता नाम |

## कमांड्स

### /salesforce:sf-status

अपने Salesforce org कनेक्शन की स्थिति, CLI संस्करण, और प्रमाणीकृत orgs
की जाँच करें।

```
/salesforce:sf-status
```

**क्या अपेक्षा करें:** आपका sf CLI संस्करण, प्रमाणीकृत org alias, उपयोगकर्ता
नाम, इंस्टेंस URL, कनेक्शन स्थिति, और API संस्करण दिखाने वाली एक तालिका।

### /salesforce:sf-login

किसी Salesforce org में प्रमाणीकरण करें। वैकल्पिक रूप से एक alias प्रदान करें।

```
/salesforce:sf-login
/salesforce:sf-login my-org
```

**क्या अपेक्षा करें:** यदि उस alias वाला org पहले से प्रमाणीकृत है, तो यह
कनेक्शन विवरण रिपोर्ट करता है। यदि नहीं, तो यह उपलब्ध क्रेडेंशियल्स
(एनवायरनमेंट वेरिएबल्स) की जाँच करता है और पहली पूर्ण रूप से संतुष्ट
विधि का उपयोग करके प्रमाणीकरण करता है।

## कौशल

### salesforce-index

शीर्ष-स्तरीय इंटेंट राउटर। जब आप प्राकृतिक भाषा में Salesforce, org
प्रबंधन, या डेटा क्वेरी का उल्लेख करते हैं तो स्वचालित रूप से सक्रिय
होता है।

| आप कहते हैं | रूट होता है |
|---------|-----------|
| "login to salesforce", "authenticate org" | `salesforce-auth` कौशल |
| "check org status", "list orgs" | `cli-operator` एजेंट |
| "show me opportunities", "look up case" | `cli-operator` एजेंट (SOQL क्वेरी) |
| "generate Apex class", "create Flow" | afv-library विकास कौशल |

### salesforce-auth

कंटेनर-अनुकूलित प्रमाणीकरण कौशल। चार प्रमाणीकरण विधियों का समर्थन करता
है और स्वचालित रूप से पता लगाता है कि कौन से क्रेडेंशियल्स उपलब्ध हैं।
प्राथमिकता: access-token, फिर JWT, फिर SFDX URL, फिर web login।

## एजेंट्स

### cli-operator

स्वायत्त Salesforce CLI एजेंट जो सुरक्षा गार्डरेल्स के साथ `sf` कमांड
निष्पादित करता है। सभी Salesforce ऑपरेशन इस एजेंट को सौंपे जाते हैं
ताकि मुख्य सेशन संदर्भ हल्का रहे।

**सुरक्षा नियम:**

- डिफ़ॉल्ट रूप से केवल-पठन — लेखन ऑपरेशनों के लिए स्पष्ट स्वीकृति आवश्यक है
- प्रीव्यू और पुष्टि चरण के बिना कभी डिप्लॉय नहीं करता
- आउटपुट में कभी भी एक्सेस टोकन, auth URLs, या क्रेडेंशियल्स प्रदर्शित नहीं करता
- सभी उपयोगकर्ता-प्रदत्त मानों को सैनिटाइज़ करता है (शेल मेटाकैरेक्टर्स को अस्वीकार करता है)
- संरचित, पार्स करने योग्य परिणामों के लिए `--json` आउटपुट का उपयोग करता है

**प्रतिक्रिया प्रारूप:**

```
## Result: [SUCCESS | FAILURE | PARTIAL]
### Command Executed
### Output Summary
### Issues
```

## उपयोग मार्गदर्शिका: खाता प्रबंधन

<Aside type="note">
  नीचे दिए गए प्रॉम्प्ट प्लगइन की हार्ड-कोडेड सुविधाएँ नहीं हैं। ये
  काम करते हैं क्योंकि cli-operator एजेंट मनमानी SOQL क्वेरी सहित कोई भी
  `sf` CLI कमांड चला सकता है। परिणाम आपके org के डेटा मॉडल और आपकी
  उपयोगकर्ता अनुमतियों पर निर्भर करते हैं। यहाँ दिए गए प्रॉम्प्ट
  मानक Salesforce ऑब्जेक्ट्स (Account, Contact, Case, Opportunity,
  AccountTeamMember, OpportunityTeamMember) के साथ परीक्षित और सत्यापित हैं।
</Aside>

ये प्रॉम्प्ट किसी भी Salesforce org के साथ काम करते हैं। प्लेसहोल्डर
मानों को अपनी जानकारी से बदलें। प्रत्येक प्रॉम्प्ट आपकी भूमिका, खातों,
और क्षेत्र के लिए विशिष्ट परिणाम उत्पन्न करता है।

<Aside type="tip">
  सर्वोत्तम परिणामों के लिए, उन प्रॉम्प्ट्स में अपना ईमेल पता शामिल करें
  जिन्हें आपके उपयोगकर्ता रिकॉर्ड की पहचान करने की आवश्यकता है। Salesforce
  orgs में समान नामों वाले कई उपयोगकर्ता हो सकते हैं।
</Aside>

### शुरुआत करना

प्रमाणीकरण के बाद, सत्यापित करें कि सब कुछ काम कर रहा है:

```
/salesforce:sf-status
```

फिर एक सरल प्राकृतिक भाषा क्वेरी आज़माएँ:

```
list my authenticated salesforce orgs
```

**क्या अपेक्षा करें:** alias, उपयोगकर्ता नाम, इंस्टेंस URL, और कनेक्शन
स्थिति के साथ सभी कनेक्टेड orgs दिखाने वाली एक तालिका।

### अपने खाते खोजें

पता लगाएँ कि आप किन खातों में असाइन हैं:

```
what salesforce accounts am I on the account team for? My email is your-email@company.com
```

**क्या अपेक्षा करें:** खाता स्वामी द्वारा समूहित खातों की एक सूची, जो
प्रत्येक पर आपकी टीम सदस्य भूमिका दिखाती है। यदि अधिकांश भूमिकाएँ खाली
दिखती हैं, तो यह एक सामान्य डेटा स्वच्छता अंतर है — खाता टीम सदस्यताएँ
मौजूद हैं लेकिन भूमिकाएँ भरी नहीं गईं।

### किसी सहकर्मी के साथ कवरेज की क्रॉस-रेफरेंस करें

अंतर खोजने के लिए किसी टीमसाथी के साथ अपने खाता कवरेज की तुलना करें:

```
In salesforce, find all accounts where Colleague Name is on the account team. Then for each of those accounts, check if your-email@company.com is also on the account team. Show me two lists: accounts where we are BOTH tagged, and accounts where only my colleague is tagged but I am missing.
```

**क्या अपेक्षा करें:** ओवरलैप काउंट और गैप काउंट दिखाने वाली एक सारांश
तालिका, इसके बाद दो सूचियाँ। यदि आपने हाल ही में भूमिकाएँ बदली हैं, तो
आपको शून्य ओवरलैप मिल सकता है — यह बताता है कि आपको किन खातों में जोड़ा
जाना चाहिए।

<Aside type="note">
  आप यह जोड़कर किसी पूर्ववर्ती की भी जाँच कर सकते हैं: "Also check if
  Previous Person Name appears on any of those accounts." यह किसी
  क्षेत्र में किसी को बदलते समय हैंडऑफ़ अंतराल की पहचान करने में मदद करता है।
</Aside>

### क्षेत्र पाइपलाइन अवलोकन

अपने खातों में सभी खुली opportunities देखें:

```
show me all open salesforce opportunities on Colleague Name's account team accounts, sorted by close date, include the stage, amount, and probability
```

**क्या अपेक्षा करें:** कुल opportunity गणना, कच्ची पाइपलाइन, और भारित
पाइपलाइन के साथ एक सारांश। फिर एक स्टेज वितरण तालिका और राशि के
अनुसार शीर्ष opportunities की सूची। आउटपुट समय-सीमा पार कर चुकी close
dates या शून्य राशि जैसे डेटा स्वच्छता मुद्दों को चिह्नित करता है।

### Opportunity गहन विश्लेषण

किसी विशिष्ट सौदे की पूरी तस्वीर प्राप्त करें:

```
show me a detailed view of the OPPORTUNITY NAME opportunity in salesforce - include the opportunity team members, any activities or tasks, and the account contacts
```

**क्या अपेक्षा करें:** Opportunity अवलोकन (stage, amount, close date,
probability, forecast category, owner), भूमिकाओं सहित opportunity टीम
सदस्य, टैग किए गए संपर्क, हाल के कार्य और इवेंट, और सौदे की राशि और
close date में समय के साथ हुए परिवर्तनों को दिखाने वाला फ़ील्ड परिवर्तन
इतिहास।

### क्षेत्र में सहायता केस

अपने खातों पर खुले सहायता केस की जाँच करें:

```
show me all open salesforce cases across Colleague Name's account team accounts, grouped by account, sorted by most recent first
```

**क्या अपेक्षा करें:** प्राथमिकता और स्थिति विभाजन के साथ खुले केस की
गणना, फिर खाते द्वारा समूहित केस विवरण। पुराने केस (बिना गतिविधि के
महीनों या वर्षों पुराने) को स्वच्छता उम्मीदवारों के रूप में चिह्नित
किया जाता है।

### तिमाही पाइपलाइन पूर्वानुमान

अपनी पाइपलाइन का पूर्वानुमान-तैयार दृश्य प्राप्त करें:

```
for Colleague Name's accounts, show me a quarterly pipeline summary - group the open opportunities by close date quarter with count, total amount, and weighted amount for each quarter
```

**क्या अपेक्षा करें:** प्रत्येक तिमाही के लिए opportunity गणना, कुल
पाइपलाइन, और भारित पाइपलाइन दिखाने वाली प्रति पंक्ति एक तिमाही वाली
तालिका। प्रति तिमाही स्टेज मिश्रण विभाजन और शीर्ष भारित सौदों को
हाइलाइट करता है। समय-सीमा पार कर चुकी opportunities को अलग से
समूहित किया जाता है।

### पूर्ण खाता अवलोकन

किसी विशिष्ट ग्राहक का गहन विश्लेषण:

```
give me a full account overview for ACCOUNT NAME in salesforce including contacts, open opportunities, and recent cases
```

**क्या अपेक्षा करें:** कंपनी प्रोफ़ाइल (उद्योग, राजस्व, कर्मचारी, स्थान),
शीर्षकों और ईमेल के साथ प्रमुख संपर्क, stages और राशियों के साथ खुली
opportunities, और स्थिति के साथ हाल के सहायता केस।

### केस लुकअप

नंबर द्वारा किसी विशिष्ट सहायता केस को देखें:

```
look up salesforce case CASE-NUMBER and show me the case details, the customer account, and who owns it
```

**क्या अपेक्षा करें:** केस विषय, स्थिति, प्राथमिकता, विवरण, ग्राहक
खाता प्रोफ़ाइल, और संपर्क जानकारी के साथ केस स्वामी।

## उपयोग मार्गदर्शिका: डेटा क्वेरी

पावर उपयोगकर्ताओं के लिए, आप प्राकृतिक भाषा में विशिष्ट SOQL-शैली क्वेरी
माँग सकते हैं। प्लगइन आपके अनुरोध को उपयुक्त SOQL में अनुवादित करता है
और उसे चलाता है।

### विशिष्ट फ़ील्ड्स

```
query salesforce for all Contacts at ACCOUNT NAME - show Name, Title, Email, Phone, and Department
```

### दिनांक फ़िल्टरिंग

```
show me all salesforce opportunities that closed won in the last 90 days on ACCOUNT NAME
```

### एकत्रीकरण

```
count all open salesforce cases grouped by priority and status across my accounts
```

### कस्टम ऑब्जेक्ट्स

```
query the CUSTOM_OBJECT__c object in salesforce for records where Status__c = 'Active'
```

<Aside type="tip">
  यदि कोई क्वेरी फ़ील्ड मौजूद न होने के कारण विफल होती है, तो प्लगइन
  स्वचालित रूप से उस फ़ील्ड के बिना पुनः प्रयास करता है। आपके org में
  कस्टम फ़ील्ड्स अलग नामों का उपयोग कर सकती हैं — यदि कोई फ़ील्ड नहीं
  मिलती तो अपने Salesforce Setup की जाँच करें।
</Aside>
