**salesforce** प्लगइन Salesforce CLI को Claude Code के साथ एकीकृत करता है,
जो org प्रमाणीकरण और एक सामान्य-उद्देश्य CLI एजेंट प्रदान करता है जो SOQL क्वेरी सहित कोई भी `sf` कमांड चला सकता है। यह JWT, access-token, और SFDX URL प्रमाणीकरण प्रवाहों के साथ headless कंटेनर परिवेशों का समर्थन करता है। नीचे दी गई उपयोग मार्गदर्शिका दर्शाती है कि CLI एजेंट के माध्यम से accounts, opportunities, cases और contacts को क्वेरी करने के लिए प्राकृतिक भाषा का उपयोग कैसे करें।

<Badge text="v1.0.0" variant="note" /> <Badge text="Development" variant="tip" />

## स्थापना

```
/plugin install salesforce@f5xc-salesdemos-marketplace
```

## पूर्वापेक्षाएँ

- **Salesforce CLI** (`@salesforce/cli`)
- **Salesforce org** API एक्सेस के साथ
- ऑब्जेक्ट क्वेरी करने की अनुमति वाला उपयोगकर्ता खाता (Cases, Opportunities, Accounts, Contacts)

अपने वर्कस्टेशन पर Salesforce CLI स्थापित करें:

```bash
brew install sf
```

स्थापना सत्यापित करें:

```bash
sf --version
```

आपको `@salesforce/cli/2.x.x` जैसा आउटपुट दिखाई देना चाहिए।

## प्रमाणीकरण सेटअप

### चरण 1: अपना Salesforce डोमेन खोजें

ब्राउज़र में अपना Salesforce URL देखें। यदि आप Salesforce को
`https://acme.lightning.force.com` पर एक्सेस करते हैं, तो आपका लॉगिन डोमेन
`https://acme.my.salesforce.com` है।

<Aside type="tip">
  यदि आपकी कंपनी SSO का उपयोग करती है, तो आपको `--instance-url` के साथ अपना
  कस्टम डोमेन निर्दिष्ट करना होगा ताकि लॉगिन सामान्य Salesforce लॉगिन पेज के
  बजाय आपके SSO प्रदाता के माध्यम से रीडायरेक्ट हो।
</Aside>

### चरण 2: ब्राउज़र के माध्यम से प्रमाणित करें

```bash
sf org login web --alias my-org --set-default --instance-url https://YOUR-DOMAIN.my.salesforce.com
```

`YOUR-DOMAIN` को चरण 1 के अपने कंपनी डोमेन से बदलें। आपका ब्राउज़र खुलेगा —
SSO या लॉगिन प्रवाह पूरा करें और ऐप को अधिकृत करें। आपको यह दिखाई देना चाहिए:

```
Successfully authorized your-email@company.com with org ID 00DXXXXXXXXXXXXXXX
```

### चरण 3: कनेक्शन सत्यापित करें

```bash
sf org display --target-org my-org
```

पुष्टि करें कि आउटपुट में आपके उपयोगकर्ता नाम और इंस्टेंस URL के साथ `Connected Status: Connected` दिखाई दे।

### चरण 4: कंटेनरों के लिए पोर्टेबल auth (वैकल्पिक)

यदि आपको headless परिवेश (CI/CD, कंटेनर, रिमोट सेशन) में प्रमाणित करने की
आवश्यकता है, तो अपने वर्कस्टेशन से SFDX auth URL निर्यात करें:

```bash
sf org display --verbose --target-org my-org
```

**Sfdx Auth Url** मान कॉपी करें (जो `force://` से शुरू होता है)। कंटेनर में, चलाएं:

```bash
export SFDX_AUTH_URL="force://PlatformCLI::YOUR_AUTH_TOKEN@your-domain.my.salesforce.com"
echo "$SFDX_AUTH_URL" | sf org login sfdx-url --sfdx-url-stdin=- --alias=my-org --set-default
```

<Aside type="caution">
  SFDX auth URL एक क्रेडेंशियल है — इसे पासवर्ड की तरह मानें। इसे फ़ाइलों में
  कमिट न करें या चैट में पेस्ट न करें। केवल एनवायरनमेंट वेरिएबल और stdin पाइप का
  उपयोग करें।
</Aside>

### प्रमाणीकरण विधियों का संदर्भ

| विधि | सर्वोत्तम उपयोग | आवश्यकता |
|--------|----------|----------|
| Web Login | ब्राउज़र वाले वर्कस्टेशन | Browser + SSO |
| SFDX URL | कंटेनर, CI/CD, पोर्टेबल auth | प्रमाणित सेशन से Auth URL |
| JWT Bearer | स्वचालित पाइपलाइन | Connected App + private key + consumer key |
| Access Token | एनवायरनमेंट वेरिएबल auth | `SF_ACCESS_TOKEN` + `SF_ORG_INSTANCE_URL` |

**नोट:** Device flow (`sf org login device`) अगस्त 2025 से ब्लॉक है और समर्थित नहीं है।

### एनवायरनमेंट वेरिएबल

| वेरिएबल | उद्देश्य |
|----------|---------|
| `SF_ACCESS_TOKEN` | access-token auth के लिए Bearer token |
| `SFDX_AUTH_URL` | SFDX URL auth के लिए Force auth URL |
| `SF_ORG_INSTANCE_URL` | Org इंस्टेंस URL |
| `SF_JWT_KEY_FILE` | JWT private key का पथ |
| `SF_CLIENT_ID` | Connected App consumer key |
| `SF_USERNAME` | JWT के लिए Salesforce उपयोगकर्ता नाम |

## कमांड

### /salesforce:sf-status

अपनी Salesforce org कनेक्शन स्थिति, CLI संस्करण और प्रमाणित orgs जांचें।

```
/salesforce:sf-status
```

**क्या अपेक्षित है:** आपके sf CLI संस्करण, प्रमाणित org एलियास, उपयोगकर्ता नाम, इंस्टेंस URL, कनेक्टेड स्थिति और API संस्करण दिखाने वाली एक तालिका।

### /salesforce:sf-login

Salesforce org को प्रमाणित करें। वैकल्पिक रूप से एक एलियास प्रदान करें।

```
/salesforce:sf-login
/salesforce:sf-login my-org
```

**क्या अपेक्षित है:** यदि उस एलियास वाला org पहले से प्रमाणित है, तो यह कनेक्शन विवरण रिपोर्ट करता है। यदि नहीं, तो यह उपलब्ध क्रेडेंशियल (एनवायरनमेंट वेरिएबल) की जांच करता है और पहली पूरी तरह संतुष्ट विधि का उपयोग करके प्रमाणित करता है।

## स्किल्स

### salesforce-index

शीर्ष-स्तरीय intent राउटर। जब आप प्राकृतिक भाषा में Salesforce, org प्रबंधन, या डेटा क्वेरी का उल्लेख करते हैं तो स्वचालित रूप से सक्रिय होता है।

| आप कहते हैं | रूट होता है |
|---------|-----------|
| "login to salesforce", "authenticate org" | `salesforce-auth` स्किल |
| "check org status", "list orgs" | `cli-operator` एजेंट |
| "show me opportunities", "look up case" | `cli-operator` एजेंट (SOQL क्वेरी) |
| "generate Apex class", "create Flow" | afv-library development स्किल्स |

### salesforce-auth

कंटेनर-अनुकूलित प्रमाणीकरण स्किल। चार auth विधियों का समर्थन करता है और स्वचालित रूप से पता लगाता है कि कौन से क्रेडेंशियल उपलब्ध हैं। प्राथमिकता: access-token, फिर JWT, फिर SFDX URL, फिर web login।

## एजेंट

### cli-operator

स्वायत्त Salesforce CLI एजेंट जो सुरक्षा गार्डरेल्स के साथ `sf` कमांड निष्पादित करता है। मुख्य सेशन संदर्भ को संक्षिप्त रखने के लिए सभी Salesforce ऑपरेशन इस एजेंट को सौंपे जाते हैं।

**सुरक्षा नियम:**

- डिफ़ॉल्ट रूप से केवल-पठन — राइट ऑपरेशन के लिए स्पष्ट अनुमोदन आवश्यक है
- पूर्वावलोकन और पुष्टि चरण के बिना कभी deploy नहीं करता
- आउटपुट में कभी access token, auth URL या क्रेडेंशियल नहीं दिखाता
- सभी उपयोगकर्ता-प्रदत्त मानों को सैनिटाइज़ करता है (shell metacharacter अस्वीकार करता है)
- संरचित, पार्स करने योग्य परिणामों के लिए `--json` आउटपुट का उपयोग करता है

**प्रतिक्रिया प्रारूप:**

```
## Result: [SUCCESS | FAILURE | PARTIAL]
### Command Executed
### Output Summary
### Issues
```

## उपयोग मार्गदर्शिका: खाता प्रबंधन

<Aside type="note">
  नीचे दिए गए प्रॉम्प्ट प्लगइन की हार्ड-कोडेड सुविधाएं नहीं हैं। ये काम
  करते हैं क्योंकि cli-operator एजेंट कोई भी `sf` CLI कमांड चला सकता है,
  जिसमें मनमानी SOQL क्वेरी शामिल हैं। परिणाम आपके org के डेटा मॉडल और
  आपकी उपयोगकर्ता अनुमतियों पर निर्भर करते हैं। यहां के प्रॉम्प्ट मानक
  Salesforce ऑब्जेक्ट (Account, Contact, Case, Opportunity,
  AccountTeamMember, OpportunityTeamMember) के साथ परीक्षित और सत्यापित हैं।
</Aside>

ये प्रॉम्प्ट किसी भी Salesforce org के साथ काम करते हैं। placeholder मानों को अपनी जानकारी से बदलें। प्रत्येक प्रॉम्प्ट आपकी भूमिका, accounts और क्षेत्र के अनुसार परिणाम देता है।

<Aside type="tip">
  सर्वोत्तम परिणामों के लिए, उन प्रॉम्प्ट में अपना ईमेल पता शामिल करें
  जिन्हें आपका उपयोगकर्ता रिकॉर्ड पहचानने की आवश्यकता है। Salesforce org में
  समान नाम वाले एकाधिक उपयोगकर्ता हो सकते हैं।
</Aside>

### शुरुआत करना

प्रमाणित करने के बाद, सत्यापित करें कि सब कुछ काम करता है:

```
/salesforce:sf-status
```

फिर एक सरल प्राकृतिक भाषा क्वेरी आज़माएं:

```
list my authenticated salesforce orgs
```

**क्या अपेक्षित है:** एलियास, उपयोगकर्ता नाम, इंस्टेंस URL और कनेक्शन स्थिति के साथ सभी कनेक्टेड orgs दिखाने वाली एक तालिका।

### अपने accounts खोजें

जानें कि आप किन accounts की account team में असाइन हैं:

```
what salesforce accounts am I on the account team for? My email is your-email@company.com
```

**क्या अपेक्षित है:** account owner द्वारा समूहीकृत accounts की एक सूची, प्रत्येक पर आपकी team member भूमिका दिखाते हुए। यदि अधिकांश भूमिकाएं खाली दिखती हैं, तो यह एक सामान्य डेटा स्वच्छता अंतराल है — account team सदस्यताएं मौजूद हैं लेकिन भूमिकाएं भरी नहीं गईं।

### किसी सहकर्मी के साथ कवरेज की तुलना करें

अंतराल खोजने के लिए किसी टीम के साथी के साथ अपनी account कवरेज की तुलना करें:

```
In salesforce, find all accounts where Colleague Name is on the account team. Then for each of those accounts, check if your-email@company.com is also on the account team. Show me two lists: accounts where we are BOTH tagged, and accounts where only my colleague is tagged but I am missing.
```

**क्या अपेक्षित है:** ओवरलैप गिनती और अंतराल गिनती दिखाने वाली एक सारांश तालिका, उसके बाद दो सूचियां। यदि आपने हाल ही में भूमिकाएं बदली हैं, तो आपको शून्य ओवरलैप मिल सकता है — यह बताता है कि आपको किन accounts में जोड़ा जाना चाहिए।

<Aside type="note">
  आप यह जोड़कर किसी पूर्ववर्ती की भी जांच कर सकते हैं: "Also check if
  Previous Person Name appears on any of those accounts." यह किसी क्षेत्र में
  किसी की जगह लेते समय हैंडऑफ अंतराल की पहचान करने में मदद करता है।
</Aside>

### क्षेत्र पाइपलाइन अवलोकन

अपने सभी accounts में सभी खुले opportunities देखें:

```
show me all open salesforce opportunities on Colleague Name's account team accounts, sorted by close date, include the stage, amount, and probability
```

**क्या अपेक्षित है:** कुल opportunity गिनती, कच्ची पाइपलाइन और भारित पाइपलाइन के साथ एक सारांश। फिर एक stage वितरण तालिका और amount के अनुसार शीर्ष opportunities की एक सूची। आउटपुट डेटा स्वच्छता समस्याओं जैसे समय-बीत-चुकी close dates या null amount को फ्लैग करता है।

### Opportunity गहन विश्लेषण

किसी विशिष्ट deal का पूरा चित्र प्राप्त करें:

```
show me a detailed view of the OPPORTUNITY NAME opportunity in salesforce - include the opportunity team members, any activities or tasks, and the account contacts
```

**क्या अपेक्षित है:** opportunity अवलोकन (stage, amount, close date, probability, forecast category, owner), भूमिकाओं के साथ opportunity team members, टैग किए गए contacts, हालिया tasks और events, और deal amount और close date के समय के साथ बदलाव दिखाने वाला field change history।

### क्षेत्र में सहायता cases

अपने accounts पर खुले support cases जांचें:

```
show me all open salesforce cases across Colleague Name's account team accounts, grouped by account, sorted by most recent first
```

**क्या अपेक्षित है:** प्राथमिकता और स्थिति विभाजन के साथ खुले cases की गिनती, फिर account द्वारा समूहीकृत case विवरण। पुराने cases (बिना किसी गतिविधि के महीनों या वर्षों पुराने) को स्वच्छता उम्मीदवारों के रूप में फ्लैग किया जाता है।

### त्रैमासिक पाइपलाइन पूर्वानुमान

अपनी पाइपलाइन का पूर्वानुमान-तैयार दृश्य प्राप्त करें:

```
for Colleague Name's accounts, show me a quarterly pipeline summary - group the open opportunities by close date quarter with count, total amount, and weighted amount for each quarter
```

**क्या अपेक्षित है:** प्रति तिमाही एक पंक्ति के साथ opportunity गिनती, कुल पाइपलाइन और भारित पाइपलाइन दिखाने वाली एक तालिका। प्रति तिमाही stage mix विवरण शामिल है और शीर्ष भारित deals को हाइलाइट करता है। समय-बीत-चुकी opportunities को अलग समूहीकृत किया जाता है।

### पूर्ण account अवलोकन

किसी विशिष्ट ग्राहक में गहराई से जाएं:

```
give me a full account overview for ACCOUNT NAME in salesforce including contacts, open opportunities, and recent cases
```

**क्या अपेक्षित है:** कंपनी प्रोफ़ाइल (उद्योग, राजस्व, कर्मचारी, स्थान), शीर्षक और ईमेल के साथ प्रमुख contacts, stage और amount के साथ खुले opportunities, और स्थिति के साथ हालिया support cases।

### Case लुकअप

नंबर द्वारा किसी विशिष्ट support case को देखें:

```
look up salesforce case CASE-NUMBER and show me the case details, the customer account, and who owns it
```

**क्या अपेक्षित है:** Case विषय, स्थिति, प्राथमिकता, विवरण, ग्राहक account प्रोफ़ाइल, और संपर्क जानकारी के साथ case owner।

## उपयोग मार्गदर्शिका: डेटा क्वेरी

पावर उपयोगकर्ताओं के लिए, आप प्राकृतिक भाषा में विशिष्ट SOQL-शैली क्वेरी मांग सकते हैं। प्लगइन आपके अनुरोध को उचित SOQL में अनुवाद करता है और चलाता है।

### विशिष्ट फ़ील्ड

```
query salesforce for all Contacts at ACCOUNT NAME - show Name, Title, Email, Phone, and Department
```

### दिनांक फ़िल्टरिंग

```
show me all salesforce opportunities that closed won in the last 90 days on ACCOUNT NAME
```

### एकत्रीकरण

```
count all open salesforce cases grouped by priority and status across my accounts
```

### कस्टम ऑब्जेक्ट

```
query the CUSTOM_OBJECT__c object in salesforce for records where Status__c = 'Active'
```

<Aside type="tip">
  यदि कोई क्वेरी किसी फ़ील्ड के मौजूद न होने के कारण विफल होती है, तो प्लगइन
  स्वचालित रूप से उस फ़ील्ड के बिना पुनः प्रयास करता है। आपके org के कस्टम
  फ़ील्ड के नाम अलग हो सकते हैं — यदि कोई फ़ील्ड नहीं मिलती तो अपना
  Salesforce Setup जांचें।
</Aside>
