**salesforce** プラグインは Salesforce CLI を Claude Code と統合し、
org 認証および SOQL クエリを含むあらゆる `sf` コマンドを実行できる汎用 CLI エージェントを提供します。JWT、アクセストークン、SFDX URL の認証フローによるヘッドレスコンテナ環境をサポートしています。以下の使用ガイドでは、CLI エージェントを通じて自然言語でアカウント、商談、ケース、連絡先をクエリする方法を説明します。

<Badge text="v1.0.0" variant="note" /> <Badge text="Development" variant="tip" />

## インストール

```
/plugin install salesforce@f5-sales-demo-marketplace
```

## 前提条件

- **Salesforce CLI** (`@salesforce/cli`)
- **Salesforce org**（API アクセス権が必要）
- オブジェクト（ケース、商談、アカウント、連絡先）をクエリするためのアクセス権を持つユーザーアカウント

ワークステーションに Salesforce CLI をインストールします：

```bash
brew install sf
```

インストールを確認します：

```bash
sf --version
```

`@salesforce/cli/2.x.x` のような出力が表示されるはずです。

## 認証のセットアップ

### ステップ 1: Salesforce ドメインを確認する

ブラウザーで Salesforce の URL を確認します。`https://acme.lightning.force.com` で Salesforce にアクセスしている場合、ログインドメインは `https://acme.my.salesforce.com` です。

<Aside type="tip">
  会社が SSO を使用している場合は、`--instance-url` でカスタムドメインを指定する必要があります。これにより、汎用の Salesforce ログインページではなく、SSO プロバイダーを経由してリダイレクトされます。
</Aside>

### ステップ 2: ブラウザーで認証する

```bash
sf org login web --alias my-org --set-default --instance-url https://YOUR-DOMAIN.my.salesforce.com
```

`YOUR-DOMAIN` をステップ 1 で確認した会社ドメインに置き換えます。ブラウザーが開くので、SSO またはログインフローを完了してアプリを承認します。以下のように表示されるはずです：

```
Successfully authorized your-email@company.com with org ID 00DXXXXXXXXXXXXXXX
```

### ステップ 3: 接続を確認する

```bash
sf org display --target-org my-org
```

出力に `Connected Status: Connected` とユーザー名およびインスタンス URL が表示されることを確認します。

### ステップ 4: コンテナ用のポータブル認証（オプション）

ヘッドレス環境（CI/CD、コンテナ、リモートセッション）で認証する必要がある場合は、ワークステーションから SFDX 認証 URL をエクスポートします：

```bash
sf org display --verbose --target-org my-org
```

**Sfdx Auth Url** の値（`force://` で始まる）をコピーします。コンテナ内で以下を実行します：

```bash
export SFDX_AUTH_URL="force://PlatformCLI::YOUR_AUTH_TOKEN@your-domain.my.salesforce.com"
echo "$SFDX_AUTH_URL" | sf org login sfdx-url --sfdx-url-stdin=- --alias=my-org --set-default
```

<Aside type="caution">
  SFDX 認証 URL は認証情報です。パスワードと同様に扱ってください。ファイルにコミットしたり、チャットに貼り付けたりしないでください。環境変数と stdin パイプのみを使用してください。
</Aside>

### 認証方式のリファレンス

| 方式 | 最適な用途 | 必要なもの |
|--------|----------|----------|
| Web ログイン | ブラウザーが利用可能なワークステーション | ブラウザー + SSO |
| SFDX URL | コンテナ、CI/CD、ポータブル認証 | 認証済みセッションの認証 URL |
| JWT ベアラー | 自動化パイプライン | コネクテッドアプリ + 秘密鍵 + コンシューマーキー |
| アクセストークン | 環境変数による認証 | `SF_ACCESS_TOKEN` + `SF_ORG_INSTANCE_URL` |

**注意:** デバイスフロー（`sf org login device`）は 2025 年 8 月以降ブロックされており、サポートされていません。

### 環境変数

| 変数 | 目的 |
|----------|---------|
| `SF_ACCESS_TOKEN` | アクセストークン認証用ベアラートークン |
| `SFDX_AUTH_URL` | SFDX URL 認証用の Force 認証 URL |
| `SF_ORG_INSTANCE_URL` | org のインスタンス URL |
| `SF_JWT_KEY_FILE` | JWT 秘密鍵へのパス |
| `SF_CLIENT_ID` | コネクテッドアプリのコンシューマーキー |
| `SF_USERNAME` | JWT 用の Salesforce ユーザー名 |

## コマンド

### /salesforce:sf-status

Salesforce org の接続ステータス、CLI バージョン、および認証済み org を確認します。

```
/salesforce:sf-status
```

**期待される出力:** sf CLI バージョン、認証済み org エイリアス、ユーザー名、インスタンス URL、接続ステータス、および API バージョンを示すテーブル。

### /salesforce:sf-login

Salesforce org に認証します。オプションでエイリアスを指定できます。

```
/salesforce:sf-login
/salesforce:sf-login my-org
```

**期待される出力:** そのエイリアスを持つ org がすでに認証済みの場合は接続詳細が表示されます。未認証の場合は、利用可能な認証情報（環境変数）を確認し、最初に条件を満たす方式で認証を行います。

## スキル

### salesforce-index

トップレベルのインテントルーター。自然言語で Salesforce、org 管理、またはデータクエリに言及すると自動的にアクティブになります。

| 発言内容 | ルーティング先 |
|---------|-----------|
| "login to salesforce"、"authenticate org" | `salesforce-auth` スキル |
| "check org status"、"list orgs" | `cli-operator` エージェント |
| "show me opportunities"、"look up case" | `cli-operator` エージェント（SOQL クエリ） |
| "generate Apex class"、"create Flow" | afv-library 開発スキル |

### salesforce-auth

コンテナ対応の認証スキル。4 つの認証方式をサポートし、利用可能な認証情報を自動検出します。優先順位：アクセストークン、次に JWT、次に SFDX URL、最後に Web ログイン。

## エージェント

### cli-operator

安全ガードレール付きで `sf` コマンドを実行する自律型 Salesforce CLI エージェント。メインセッションのコンテキストを軽量に保つため、すべての Salesforce 操作はこのエージェントに委譲されます。

**安全ルール：**

- デフォルトは読み取り専用 — 書き込み操作には明示的な承認が必要
- プレビューと確認ステップなしにはデプロイを実行しない
- アクセストークン、認証 URL、または認証情報を出力にエコーしない
- すべてのユーザー提供の値をサニタイズする（シェルメタキャラクターを拒否）
- 構造化された解析可能な結果のために `--json` 出力を使用

**レスポンス形式：**

```
## Result: [SUCCESS | FAILURE | PARTIAL]
### Command Executed
### Output Summary
### Issues
```

## 使用ガイド：アカウント管理

<Aside type="note">
  以下のプロンプトはプラグインのハードコードされた機能ではありません。cli-operator エージェントが任意の SOQL クエリを含むあらゆる `sf` CLI コマンドを実行できるため動作します。結果は org のデータモデルとユーザーのアクセス権によって異なります。ここに記載されたプロンプトは、標準的な Salesforce オブジェクト（Account、Contact、Case、Opportunity、AccountTeamMember、OpportunityTeamMember）で動作することがテストおよび検証されています。
</Aside>

これらのプロンプトはどの Salesforce org でも機能します。プレースホルダーの値を自分の情報に置き換えてください。各プロンプトは、あなたのロール、アカウント、およびテリトリーに固有の結果を生成します。

<Aside type="tip">
  最良の結果を得るために、ユーザーレコードを特定する必要があるプロンプトにはメールアドレスを含めてください。Salesforce org には類似した名前のユーザーが複数存在する場合があります。
</Aside>

### はじめに

認証後、すべてが正常に動作することを確認します：

```
/salesforce:sf-status
```

次に、シンプルな自然言語クエリを試します：

```
list my authenticated salesforce orgs
```

**期待される出力:** エイリアス、ユーザー名、インスタンス URL、および接続ステータスを示すすべての接続済み org のテーブル。

### 自分のアカウントを確認する

自分がアカウントチームに割り当てられているアカウントを確認します：

```
what salesforce accounts am I on the account team for? My email is your-email@company.com
```

**期待される出力:** アカウントオーナー別にグループ化されたアカウントのリスト。各アカウントにおけるチームメンバーのロールを表示します。ほとんどのロールが空の場合、それはよくあるデータ品質の問題です — アカウントチームのメンバーシップは存在しますが、ロールが設定されていません。

### 同僚とのカバレッジを相互確認する

チームメートとのアカウントカバレッジを比較してギャップを見つけます：

```
In salesforce, find all accounts where Colleague Name is on the account team. Then for each of those accounts, check if your-email@company.com is also on the account team. Show me two lists: accounts where we are BOTH tagged, and accounts where only my colleague is tagged but I am missing.
```

**期待される出力:** 重複数とギャップ数を示すサマリーテーブルと、2 つのリスト。最近ロールを変更した場合、重複がゼロになることがあります — これは追加が必要なアカウントを明らかにします。

<Aside type="note">
  「Also check if Previous Person Name appears on any of those accounts.」を追加して、前任者を確認することもできます。これにより、テリトリーの引き継ぎ時の handoff ギャップを特定するのに役立ちます。
</Aside>

### テリトリーパイプラインの概要

アカウント全体のすべてのオープン商談を確認します：

```
show me all open salesforce opportunities on Colleague Name's account team accounts, sorted by close date, include the stage, amount, and probability
```

**期待される出力:** 商談の総数、未重み付けパイプライン、重み付けパイプラインを含むサマリー。次にステージ分布テーブルと金額上位の商談リスト。期限超過のクローズ日や金額が null の場合など、データ品質の問題にフラグが立てられます。

### 商談の詳細分析

特定の案件の全体像を把握します：

```
show me a detailed view of the OPPORTUNITY NAME opportunity in salesforce - include the opportunity team members, any activities or tasks, and the account contacts
```

**期待される出力:** 商談の概要（ステージ、金額、クローズ日、確度、予測カテゴリ、オーナー）、ロールを含む商談チームメンバー、タグ付けされた連絡先、最近のタスクとイベント、案件金額とクローズ日の推移を示すフィールド変更履歴。

### テリトリー全体のサポートケース

アカウントのオープンサポートケースを確認します：

```
show me all open salesforce cases across Colleague Name's account team accounts, grouped by account, sorted by most recent first
```

**期待される出力:** 優先度とステータスの内訳を含むオープンケースの数と、アカウント別にグループ化されたケースの詳細。活動のない古いケース（数か月または数年前）はデータ品質の候補としてフラグが立てられます。

### 四半期パイプライン予測

パイプラインの予測対応ビューを取得します：

```
for Colleague Name's accounts, show me a quarterly pipeline summary - group the open opportunities by close date quarter with count, total amount, and weighted amount for each quarter
```

**期待される出力:** 四半期ごとに商談数、パイプライン総額、重み付けパイプラインを示すテーブル。各四半期のステージ混在内訳と上位の重み付け案件がハイライトされます。期限超過の商談は別途グループ化されます。

### アカウントの全体概要

特定の顧客の詳細を確認します：

```
give me a full account overview for ACCOUNT NAME in salesforce including contacts, open opportunities, and recent cases
```

**期待される出力:** 企業プロフィール（業種、売上、従業員数、所在地）、役職とメールを含む主要連絡先、ステージと金額を含むオープン商談、ステータスを含む最近のサポートケース。

### ケース照会

ケース番号で特定のサポートケースを検索します：

```
look up salesforce case CASE-NUMBER and show me the case details, the customer account, and who owns it
```

**期待される出力:** ケースの件名、ステータス、優先度、説明、顧客アカウントのプロフィール、および連絡先情報を含むケースオーナー。

## 使用ガイド：データクエリ

上級ユーザー向けに、自然言語で SOQL スタイルのクエリを要求できます。プラグインがリクエストを適切な SOQL に変換して実行します。

### 特定のフィールド

```
query salesforce for all Contacts at ACCOUNT NAME - show Name, Title, Email, Phone, and Department
```

### 日付フィルタリング

```
show me all salesforce opportunities that closed won in the last 90 days on ACCOUNT NAME
```

### 集計

```
count all open salesforce cases grouped by priority and status across my accounts
```

### カスタムオブジェクト

```
query the CUSTOM_OBJECT__c object in salesforce for records where Status__c = 'Active'
```

<Aside type="tip">
  フィールドが存在しないためクエリが失敗した場合、プラグインはそのフィールドを除いて自動的に再試行します。org 内のカスタムフィールドは異なる名前を使用している場合があります — フィールドが見つからない場合は Salesforce の設定を確認してください。
</Aside>
